# Nacre — Codebase Deep Dive & Improvement Catalog

**Date:** 2026-06-13
**Scope:** Full monorepo (`packages/core`, `parser`, `cli`, `viz`, `dashboard`, `sdk`; ~24k LOC)
**Status of code reviewed:** `main` @ `aebfc31`

## How this review was produced

A multi-agent review: seven parallel reviewers each took one dimension (architecture,
security, performance, code quality, UI/UX, documentation, testing) and grounded every
finding in real `file:line` evidence. Each finding was then handed to an **independent
adversarial verifier** that re-opened the cited source and judged it `confirmed`,
`false_positive`, `mischaracterized`, or `needs_nuance` — correcting severities where the
reviewer over- or under-stated impact. A completeness critic then hunted for what the seven
dimensions missed (concurrency, data migration, packaging, data-egress privacy, embedding
lifecycle). A separate pass proposed forward-looking features.

**Result:** 66 dimension findings (54 confirmed, 12 confirmed-with-nuance, **0 false
positives**) + 6 critic findings + 31 feature ideas. Severities below are the *verifier-corrected*
values; where the verifier changed a rating or added important caveats, a "Verification detail"
toggle is included.

> Severity uses real-world impact, not theoretical: `critical` = broken/exploitable now,
> `high` = serious and likely to bite, `medium` = real but bounded, `low`/`info` = polish.

## Executive summary

The **core engine is genuinely well-built**, and the weaknesses are almost entirely at the
**edges and seams**. The project sprinted through ~11 milestones (SQLite, embeddings, MCP,
REST API, SDK, hive consolidation, a React dashboard) while the **tooling, documentation, and
integration glue never caught up**.

**Strengths (keep these):**
- Clean, acyclic core module graph; a real `GraphStore` interface; coherent data model in `types.ts`.
- TypeScript strict mode genuinely on across all packages; only 3 `as any`, no `@ts-ignore`.
- SQL is consistently parameterized (no injection); no committed secrets; API keys from env, not logged.
- Test *content* is strong: ~7k LOC of edge-case-aware tests (decay half-life tables, store
  serialization roundtrips, mocked embedders, entity dedup).
- The CLI is the best-finished surface: consistent `citty` subcommands, actionable errors, broad `--format json`.

**Themes that need attention:**
1. **Two surfaces are silently broken** — the React dashboard does not compile; MCP semantic recall returns noise.
2. **The network API is an open door** — no auth, binds `0.0.0.0`, wildcard CORS, path traversal on `/consolidate`.
3. **The foundation is unguarded** — no CI, no linter, test runners that fail, native-module version drift.
4. **Hot paths don't scale** — recall rescans the whole graph + brute-force vector scan + N+1; consolidation rewrites everything.
5. **Docs describe a different (earlier) product** — "Phase 1 / JSON persistence" vs the shipped SQLite/MCP/API reality.

## Tier 1 — what to fix first, by theme

### A. Surfaces that are silently broken
- **The React dashboard does not compile** (`UX-1`, CRITICAL). `useMemo` unimported, `<TimeScrubber>` undefined. `nacre dashboard`'s first run is a blank/error screen. The deeper fix is CI/typecheck on commit.
- **MCP semantic recall returns noise** (`SEC`/feature). `mcp/tools.ts:39` hardcodes the mock embedder; mock vectors are matched against a real index → dimension mismatch → empty/graph-only. And `nacre_remember` writes nodes without embedding them, so agent-written memories are invisible to semantic recall.
- **Embedding provider/dimension bookkeeping is broken on the main path** (`CRIT-2`). The recall guard reads a meta key only `nacre embed` writes; the consolidate/ingest pipeline never records provider/dims → switching providers silently yields empty results.

### B. Security — the network API is an open door
- No auth + binds `0.0.0.0` (`SEC-1`), wildcard CORS (`SEC-2`), arbitrary-path read/write via `POST /consolidate` (`SEC-3`), no rate/size limits (`SEC-4`), raw error leakage (`SEC-5`). For an agent's *private memory store*, this is confidentiality + integrity (poisoning) + availability.

### C. The foundation is unguarded
- `npm test` is a false green (`TEST-1`, `TEST-2`): root runs core only; cli/parser `node --test` fail 100%.
- No linter, no CI (`QUAL-2`) — this is *why* a non-compiling dashboard and `as any` casts reached `main`.
- Native-module version drift (`QUAL-3`, `TEST-3`): `better-sqlite3` ^11 (core) vs ^12 (root) installed **both** binaries; `zod` ^3 vs ^4 coexist.
- `SQLITE_BUSY` waiting to happen (`CRIT-1`): no `busy_timeout`, yet CLI+API+MCP all open the same DB.

### D. Performance hot paths (all on the live server path)
- `recall()` loads the entire graph per query (`PERF-3`), brute-force vector scan twice per query (`PERF-4`), then N+1 queries (`PERF-5`); consolidation rewrites the whole graph + full snapshot every run (`PERF-6`); frontends never dispose GPU/listeners/RAF (`PERF-7`).

### E. Architecture debt (high, but not an active bug)
- Two forked WebGL frontends drifting apart (`UX-2`, `ARCH-1`); the data model hand-redefined across frontends/SDK (`ARCH-2`).

A concrete sequencing recommendation is at the end of this document.

## Full findings catalog


### Architecture & Code Structure

_Core's internal module graph is clean and acyclic (everything flows down to types/graph/decay; store is a near-leaf, embeddings has no upward deps), and there is a genuine `GraphStore` interface in store.ts — so the bones are good. But the abstraction is undermined in practice (every consumer, including core's own ingest.ts, types against the concrete `SqliteStore`), the two web frontends (viz vanilla + dashboard React) are forked copies of the same 3d-force-graph rendering code that have silently diverged, and the data model is hand-redefined in 5+ places with no shared types/contract. Dependency versions drift heavily across packages (better-sqlite3 11 vs 12, zod 3 vs 4, hono 4.7 vs 4.11), and the REST API server lives inside the CLI package, forcing deep cross-package source imports. These are maintainability/correctness hazards rather than vision problems._

#### ARCH-2 · [HIGH] Hard runtime-dependency version drift across packages (better-sqlite3, zod, hono)
- **File:** `packages/core/package.json:15 (cf. package.json:43)`  
- **Effort:** small  
- **Verdict:** confirmed (high confidence)

Multiple shared runtime deps are pinned to incompatible major/minor ranges in different package.jsons. core declares better-sqlite3 ^11.0.0 while the root declares ^12.6.2 (and @types/better-sqlite3 ^7) — a native module where 11 vs 12 differ in prebuilt binaries / Node ABI, which is exactly the kind of mismatch that breaks at install/load time. cli pins zod ^3.24.4 while the root pins zod ^4.3.6 (zod 3→4 is a breaking API change). hono is ^4.7.6 in cli/sdk vs ^4.11.7 at root, and @hono/node-server ^1.13.8 vs ^1.19.9. No package declares an `engines.node` field and there is no .nvmrc/volta, so the noted Node version drift is unconstrained.

Evidence:

~~~
core/package.json:15 `"better-sqlite3": "^11.0.0"` vs package.json:43 `"better-sqlite3": "^12.6.2"`; cli/package.json:18 `"zod": "^3.24.4"` vs package.json:46 `"zod": "^4.3.6"`
~~~

**Fix:** Unify versions: hoist shared deps to a single source of truth (root + workspace ranges that agree), align better-sqlite3 to one major across root and core, pick zod 3 OR 4 repo-wide, and add `engines.node` + an .nvmrc to pin a Node version. Consider syncpack to enforce this in CI.

#### ARCH-4 · [MEDIUM] GraphStore interface exists but every consumer (incl. core itself) couples to concrete SqliteStore
- **File:** `packages/core/src/store.ts:302-373`  
- **Effort:** medium  
- **Verdict:** confirmed (high confidence) _(reviewer rated **high**, verifier corrected to **medium**)_

store.ts defines a clean `GraphStore` interface (~70 methods) that `SqliteStore implements`, suggesting the persistence layer is swappable. In reality nothing depends on the interface: core's own ingest.ts imports the concrete class (`import type { SqliteStore } from './store.js'`), and every CLI/API/MCP consumer types against `SqliteStore` directly (graph-loader.ts, mcp/tools.ts, mcp/resources.ts, and all 9 api/routes/*.ts take `store: SqliteStore`). Because construction is `SqliteStore.open(path)` (a static on the concrete class) with no factory returning the interface, the abstraction provides zero substitutability and is effectively dead code that has to be kept in sync with the impl by hand.

Evidence:

~~~
ingest.ts:8 `import type { SqliteStore } from './store.js';`  api/routes/graph.ts:4 `export function graphRoutes(store: SqliteStore): Hono`  api/server.ts:3 `import type { SqliteStore } from '@nacre/core';`
~~~

**Fix:** Either commit to the abstraction — have consumers (especially api/routes and ingest) depend on `GraphStore`, add an `openStore(path): GraphStore` factory, and keep `SqliteStore` internal — or delete the interface and stop paying the dual-maintenance cost. Given the size, depend-on-interface is the better long-term call since the API layer is the natural seam.

#### ARCH-6 · [LOW] store.ts is a 1264-line god-object spanning 7 unrelated persistence domains
- **File:** `packages/core/src/store.ts:302-373`  
- **Effort:** large  
- **Verdict:** needs nuance (high confidence) _(reviewer rated **medium**, verifier corrected to **low**)_

SqliteStore mixes graph nodes/edges, file hashes, embeddings + vector similarity search, episodes + episode-entity links, procedures, snapshots/temporal history, and meta key-value — ~70 methods in one class and one ~70-method interface. The single GraphStore interface forces every consumer (e.g. the graph-only api routes, or the embeddings-only embed command) to depend on the entire surface. Evidence:

~~~
store.ts:335-344 episode methods, :347-351 procedure methods, :354-360 snapshot/history methods, :324-332 embedding/similarity methods — all on one `SqliteStore implements GraphStore`
~~~

**Fix:** Split into focused stores/interfaces sharing one Database handle: GraphStore (nodes/edges/files), EmbeddingStore, EpisodeStore, ProcedureStore, SnapshotStore. Compose them (a facade or DI) so consumers depend only on the slice they use. Move the SQL DDL/schema into a migrations module separate from query methods.

<details><summary>Verification detail</summary>

The file and class description are accurate but the method count is overstated, and the practical impact claim about consumer coupling is the opposite of the truth.

**What is accurate:**
- store.ts is exactly 1264 lines (confirmed by wc -l).
- The `GraphStore` interface spans lines 302-373 and covers 10 distinct domain sections (Node, Edge, File tracking, Embedding, Episode, Procedure, Snapshot, Bulk, Metadata, Lifecycle — visible via the comment headers at lines 303, 311, 318, 323, 334, 346, 353, 362, 366, 370).

**What is overstated or wrong:**
1. Method count: the finding says "~70 methods in one class and one ~70-method interface." The actual count is 51 public methods on both the interface and class (plus 3 privates). The "~70" figure is off by ~40%.
2. Consumer coupling claim inverted: The finding asserts "the single GraphStore interface forces every consumer... to depend on the entire surface." **Severity correction:** The finding rates this "medium." Given that (a) the interface is unused as a decoupling mechanism anyway, (b) this is a typical SQLite adapter pattern, and (c) the file is 1264 lines of straightforward CRUD that navigates easily by section headers, real-world impact is low. A motivated engineer could extract sub-stores, but it would not unblock any current pain point. The correct severity is "low."

</details>

### Security

_The Nacre REST API and MCP server have no authentication, authorization, or rate limiting of any kind, and the API binds to 0.0.0.0 by default with fully-open CORS — so any host on the network (and, via the wildcard CORS, any malicious web page the user visits) can read, create, and delete the agent's entire memory graph. The most serious concrete bug is the POST /consolidate endpoint, which accepts arbitrary filesystem paths and reads them, giving unauthenticated arbitrary-file ingestion/traversal. SQL is consistently parameterized (no injection), no secrets are committed, and API keys are sourced from env vars and not logged — those areas are clean. Remaining issues are prompt injection from ingested content into the LLM extractor, raw error-message leakage to clients, and minor local-only shell interpolation in the viz command._

### Performance & Scalability

_Nacre's core read/write paths are built around loading and rescanning the entire graph in memory on nearly every operation, so cost scales superlinearly with graph size. The two highest-impact problems are O(files × entities × nodes) entity resolution during consolidation (with redundant per-node string normalization) and a recall path that exports the full graph plus does a brute-force linear cosine scan over every stored embedding, then issues N+1 store queries per candidate. The consolidation write path also rewrites the entire graph (importGraph DELETE+reinsert) and then snapshots a full copy of every node/edge on every run, doubling/tripling write volume. Both web frontends leak Three.js GPU resources and event listeners (zero dispose/removeEventListener/cancelAnimationFrame calls) and recreate all node meshes on every hover._

#### PERF-1 · [HIGH] recall() exports the entire graph into memory on every query
- **File:** `packages/core/src/recall.ts:163-167`  
- **Effort:** large  
- **Verdict:** confirmed (high confidence)

Every recall() call invokes store.getFullGraph(), which calls listNodes() and listEdges() (each a full-table SELECT) and materializes Record&lt;id,node&gt;/Record&lt;id,edge&gt; for the whole graph, plus JSON.parse of aliases/sourceFiles/excerpts/evidence for every row. graphWalk() then calls buildAdjacencyMap(graph) which iterates all nodes and edges again to build the adjacency map from scratch. So each query is O(nodes + edges) just to set up, regardless of how few candidates the query touches. For a server (Hono API/MCP) handling many recall calls this re-deserializes the full graph per request.

Evidence:

~~~

    graph = store.getFullGraph();
  }
  ...

**Fix:** Avoid loading the full graph per query. Seed from semantic/fuzzy hits, then expand the graph walk using targeted store.listEdges({source}/{target}) (indexed) for only the frontier nodes, fetching node rows on demand and caching them. Alternatively cache the parsed graph + adjacency map in the store and invalidate on writes. For a long-lived server this is the single biggest win.

## Cross-cutting findings (completeness critic)

_Issues the seven dimensions missed: concurrency, data migration, packaging, LLM truncation, data-egress privacy._

<a id="crit-1"></a>
#### CRIT-1 · [HIGH] SQLite store has no busy_timeout/retry while CLI, API, and MCP all open the same DB concurrently (SQLITE_BUSY)
- **File:** `packages/core/src/store.ts:402-410`  
- **Effort:** trivial

SqliteStore.open() enables WAL + synchronous=NORMAL but never sets a busy_timeout pragma, and there is no SQLITE_BUSY retry/backoff anywhere (grep for busy_timeout and retry both return nothing). Every entry point opens its own connection to the same .db file via SqliteStore.open(): the API server (commands/api.ts:40), the MCP server (mcp/server.ts:11), and ~15 one-shot CLI commands (graph-loader.ts, recall.ts, ingest.ts, hive.ts, etc.). The moment two of these touch the graph at once — e.g. the long-running API server is up while the user runs `nacre consolidate` or `nacre embed`, or two MCP mutation tools fire — better-sqlite3 will throw `SQLITE_BUSY: database is locked` immediately with the default 0ms timeout. This is a data-availability bug for the intended multi-surface usage (REST + MCP + CLI on one graph) and was not raised by any architecture/performance/quality finding.

Evidence:

~~~
db.pragma('journal_mode = WAL');
    db.pragma('synchronous = NORMAL');
    db.pragma('foreign_keys = ON');  // no db.pragma('busy_timeout = ...') and no retry logic exists
~~~

**Fix:** Add `db.pragma('busy_timeout = 5000')` (or higher) in open(), and consider a single shared store process or a write queue. At minimum, document that the API/MCP server must not run concurrently with mutating CLI commands.

<a id="crit-2"></a>
#### CRIT-2 · [HIGH] Pipeline writes embeddings without recording embedding_dimensions/provider, making the recall mismatch guard dead and provider switches silently break semantic recall
- **File:** `packages/core/src/recall.ts:117-122`  
- **Effort:** small

recall() guards against a provider/model change by reading the `embedding_dimensions` meta key, but that key is ONLY written by the standalone `nacre embed` command (commands/embed.ts:150). The consolidate/ingest pipeline writes embeddings directly via store.putEmbedding (parser/pipeline.ts:247,261 and core/ingest.ts:193) and never sets embedding_dimensions, nor does it call setEmbeddingProvider() (which is what clears stale vectors on a provider switch, store.ts:671-678). Consequences: (a) after `nacre consolidate --provider X`, the recall dimension-mismatch warning at recall.ts:117 is dead because storedDims is undefined; (b) if the user later consolidates/recalls with a different provider (different dimensions), searchSimilar silently skips every row via `if (vec.length !== query.length) continue` (store.ts:726), returning empty semantic results with no error and no warning; (c) mixed-provider vectors can coexist in one table. The embeddings model lifecycle is exactly the cross-cutting concern the review was asked to look for and it is broken across the main ingestion path.

Evidence:

~~~
recall.ts:117  const storedDims = store.getMeta('embedding_dimensions');
...
pipeline.ts:247  store.putEmbedding(node.id, 'node', text, vector, provider.name); // never store.setMeta('embedding_dimensions', ...) nor setEmbeddingProvider()
~~~

**Fix:** In the pipeline/ingest embedding loops, call store.setEmbeddingProvider(provider.name) and store.setMeta('embedding_dimensions', String(provider.dimensions)) before/after embedding, so the provider-switch clear and the recall mismatch guard actually work. Better: move dimension/provider bookkeeping into putEmbedding itself.

<a id="crit-3"></a>
#### CRIT-3 · [MEDIUM] Schema migration has no upgrade guard for newer-than-code DBs and silently rewrites the version backward
- **File:** `packages/core/src/store.ts:417-515`  
- **Effort:** small

The migration block only handles `ver < N` upgrades. If a graph .db was written by a newer binary (on-disk schema_version &gt; code SCHEMA_VERSION=5) — e.g. a teammate or a future release opened it — none of the `if (ver < n)` branches run, the store opens against a schema it doesn't understand, and then line 514 unconditionally does `INSERT OR REPLACE ... schema_version = 5`, silently stamping the newer DB back down to the older code's version. There is no check like `if (ver > SCHEMA_VERSION) throw`. This is the data-migration/versioning concern the review was asked to surface: a downgrade either corrupts version tracking or operates on incompatible columns with no error. The migrate.ts JSON importer (commands/migrate.ts:45) likewise JSON.parses a NacreGraph with zero version/shape validation.

Evidence:

~~~
if (ver < 5) { ... }
      db.prepare("INSERT OR REPLACE INTO meta (key, value) VALUES (?, ?)").run('schema_version', String(SCHEMA_VERSION)); // runs even when ver > SCHEMA_VERSION
~~~

**Fix:** Before applying migrations, add `if (ver > SCHEMA_VERSION) throw new Error('Graph was created by a newer nacre version (schema ' + ver + '); upgrade nacre.')`. Only stamp schema_version forward, never down. Validate the imported JSON shape in migrate.ts.

<a id="crit-4"></a>
#### CRIT-4 · [MEDIUM] Packages are unpublishable: file: workspace deps leak into manifests, and per-package license/private fields are missing
- **File:** `packages/cli/package.json:14-15`  
- **Effort:** small

@nacre/cli and @nacre/sdk depend on sibling packages via `file:../core` / `file:../parser` specifiers rather than `workspace:*` or real versions. These resolve locally but are written verbatim into the published manifest, so `npm install @nacre/cli` from the registry would request `@nacre/core@file:../core` and fail — the packages cannot be released as-is. Compounding this: the repo LICENSE is Apache-2.0 and root package.json declares it, but none of the publishable packages (core/parser/cli/sdk/viz) carry a `license` field, and only dashboard is marked `private: true` — so core/parser/cli/sdk/viz are implicitly publishable with no license metadata and no `files` allowlist in some cases. There is no prepublishOnly/publishConfig/.npmignore anywhere. This release/packaging dimension was not covered by any finding.

Evidence:

~~~
cli/package.json:14  "@nacre/core": "file:../core",
  "@nacre/parser": "file:../parser",
# and: per-package grep for \"license\" returns only root package.json (Apache-2.0); core/parser/cli/sdk/viz have none
~~~

**Fix:** Switch internal deps to `workspace:*` (or `^0.x` matching real versions) and rely on the publish tool to rewrite them; add `"license": "Apache-2.0"` to every package.json, mark non-published packages `private: true`, and add a release config (publishConfig/prepublishOnly).

<a id="crit-5"></a>
#### CRIT-5 · [MEDIUM] Anthropic entity extractor sends whole note in one max_tokens=4096 call and ignores stop_reason, so large notes silently extract zero entities
- **File:** `packages/parser/src/extract/llm-extractor.ts:29-46`  
- **Effort:** medium

createAnthropicProvider() puts the entire concatenated note into a single message with `max_tokens: 4096` and never inspects `stop_reason`/`usage` on the response, nor does buildPrompt chunk or bound input size (it joins ALL sections). For a large memory file the model's JSON array gets truncated at the token cap; parseEntityResponse then fails JSON.parse and returns [] with only a console.warn (llm-extractor.ts:118-121). The net effect is that the bigger and more entity-rich a note is, the more likely it silently yields zero entities — the opposite of intent — with no surfaced error and wasted API spend. This is both a cost/telemetry concern (no usage tracking) and a correctness concern (silent truncation) that the LLM-extraction findings (only flagged: untested + unreachable + prompt-injection) did not cover. Note: per the claude-api model conventions, `claude-haiku-4-5` is a valid current alias, so the model id itself is fine.

Evidence:

~~~
max_tokens: 4096,
        messages: [{ role: 'user', content: prompt }],
...
} catch { console.warn(`[llm-extractor] Failed to parse LLM response as JSON: ...`); return []; }  // no stop_reason check, no chunking
~~~

**Fix:** Check `data.stop_reason === 'max_tokens'` and warn/retry or chunk; bound/segment the input per section instead of one mega-prompt; raise max_tokens or paginate; record `usage` for cost visibility.

<a id="crit-6"></a>
#### CRIT-6 · [MEDIUM] Private memory content is sent verbatim to external OpenAI/Ollama/Anthropic endpoints with no opt-in gate or redaction
- **File:** `packages/core/src/embeddings.ts:271-279`  
- **Effort:** medium

Node excerpts and full episode content (the agent's private memories) are transmitted in cleartext to third-party services with no consent gate, no allow/deny config, and no redaction: OpenAIEmbedder.embedBatch POSTs raw text to api.openai.com (embeddings.ts:272-279), OllamaEmbedder POSTs to a configurable baseUrl (embeddings.ts:200-204), and the LLM extractor POSTs whole note bodies to api.anthropic.com (llm-extractor.ts:22-33). The default embedding provider is even returned as 'ollama' (store.ts:668), and resolveProvider/config (config.ts) provide no way to declare 'this graph must stay local-only'. For a product whose entire purpose is storing an agent's private memory, the absence of any telemetry/privacy boundary around what leaves the machine is a notable gap the security review (which focused on the inbound API surface: auth, CORS, path traversal, prompt injection) did not address from the data-egress side.

Evidence:

~~~
embeddings.ts:272  const res = await fetch(`${this.baseUrl}/embeddings`, { ... body: JSON.stringify({ model: this.model, input: texts }) });
store.ts:668  return this.getMeta('embedding_provider') ?? 'ollama';  // default sends content off-box
~~~

**Fix:** Add an explicit local-only/offline mode (e.g. config flag that restricts providers to onnx/mock and refuses network embedders), default to the local ONNX provider, surface a one-time warning when content first leaves the machine, and document the data-egress behavior of each provider.

## Feature recommendations

The 31 raw proposals (full table in the appendix) deduplicate into the following. Every item
builds on code that already exists.

### Big bets — the differentiators

The embedding and temporal infrastructure is **already built and largely idle**; these turn it
into the product Nacre's vision describes.

| Feature | Why now | Builds on |
|---|---|---|
| **Semantic emergent-edge discovery** | The single biggest unrealized differentiator. `searchSimilar` is used at recall time but never to *create* edges — memories about the same thing in different words never link. Add a consolidation pass proposing threshold-gated, decay-subject `semantic` edges. | `store.searchSimilar`, pipeline embed loop, `generateSuggestions` |
| **Graph-native path reasoning** ("how is X connected to Y?") | CONCEPT's killer example is a multi-hop path, but there is no `findPath` anywhere. Weighted Dijkstra over `-log(weight)`; expose as `nacre path`, `nacre_relate` MCP tool, REST. Turns the graph from an index into a reasoning surface. | `buildAdjacencyMap`, `computeCurrentWeight`, `graphWalk` |
| **Automated procedure mining** | Procedural memory is the stated key differentiator and is fully modeled + manually writable, but nothing auto-extracts it. Mine recurring decision/preference episodes into draft (`flaggedForReview`) procedures. | `episode-extractor` DECISION_RE, `procedures.ts`, `llm-extractor` |
| **Sleep-cycle replay & reconsolidation** | Consolidate only touches new/changed files then decays — no replay over existing memory. Add spreading reinforcement, **real** dormant-edge pruning (today decay only flags), and late near-duplicate merge (hive merge logic is reusable). | `decayAllEdges`, `hive.ts` merge, `reinforceNode/Edge` |
| **Contradiction & stale-belief detection** | `contradictions` counter + auto-flag exist but are only bumped by manual negative feedback — nothing *detects* a new memory conflicting with an old procedure. | `applyProcedure`, `findRelevantProcedures`, LLM-as-judge |
| **Entity evolution / timeline + `asOf` over MCP** | VISION promises "how has my understanding of X changed?" — temporal infra (`getNodeHistory`, snapshots, `asOf`) is built but only surfaced as graph-level diffs. | `temporal.ts`, `getNodeHistory`, recall `asOf` |
| **Importance-weighted decay** | `accessCount`/`lastAccessed`/`importance`/procedure `sourceNodes` are collected but inert. Wire them into `computeCurrentWeight` for real "smart forgetting." | `decay.ts`, recall importance term, `nacre_feedback` |
| **Consolidation animation** ("watch the sleep cycle") | The README-selling GIF the roadmap calls for; time-scrub + per-date weights + `transition.ts` already exist. | `computeWeightAtDate`, `transition.ts`, `diffSnapshots` |
| **Decay experiment runner & metrics** | `docs/EXPERIMENTS.md` is a fully-designed, entirely unbuilt research program. `GraphConfig` already parameterizes the exact knobs. Lets the project validate its central thesis quantitatively. | `consolidate(config)`, `GraphConfig`, weighted community detection |

### Quick wins (small, high-leverage)

- **`nacre stats` / `--doctor`** — counts, provider/dims, health flags (orphans, unembedded, dim mismatch); all counters already exist.
- **`NACRE_GRAPH` env var + shared `graphArg` helper** — end the per-command `--graph` inconsistency.
- **`consolidate` ergonomics** — variadic sources, `--ignore`, `--provider`/`--llm` (pipeline already accepts these; this is also how you expose the currently-unreachable LLM extractor).
- **`consolidate --dry-run` + `--format json`** — the one mutating command with no preview/JSON.
- **`nacre init`** — scaffold graph dir + `nacre.config.json` (no template ships today) + entity-map.
- **Friendlier "needs a SQLite graph" error** — 8 commands dead-end on a JSON graph; auto-suggest the `nacre migrate` command.
- **Expand MCP tools** — `nacre_episodes`, `nacre_insights`, `nacre_alerts`, `nacre_consolidate` are thin wrappers over existing core/REST functions invisible to agents today.
- **Narrated memory diff** — turn the raw `GraphDiff` into "Emerged / Crystallized / Fading" by running `analyzeSignificance` across two snapshots.
- **CLI color** (picocolors, gated on `isTTY`/`NO_COLOR`).

### Foundation (makes everything else safe)

`@nacre/contracts` shared types · route everything through `GraphStore` + an in-memory test store ·
CI + Biome lint/format · single-version deps + `engines` · structured injectable logger ·
authenticated/hardened API mode · consolidate the two frontends into `@nacre/graph-ui`.

## Suggested sequencing

1. **Stop the bleeding (days):** CI + Biome + fix the test runners (catches the dashboard/`as any`/drift class of bug), then fix the dashboard compile, MCP mock-embedder + auto-embed, `busy_timeout`, and embedding-provider bookkeeping.
2. **Close the security gap (days):** default `127.0.0.1`, bearer token, CORS allowlist, confine `/consolidate`, sanitize errors; add a local-only egress mode.
3. **Unify the foundation (1–2 wk):** `@nacre/contracts`, single-version deps, `GraphStore` everywhere + in-memory store, docs truth-up (CLAUDE.md / README / ARCHITECTURE / ROADMAP; delete scratch files).
4. **Performance pass (1–2 wk):** cache graph+adjacency on the server, pre-normalize vectors, kill recall N+1, delta consolidation writes, frontend disposal.
5. **Then the differentiators:** semantic emergent edges → path reasoning → procedure mining → reconsolidation → experiment runner.
